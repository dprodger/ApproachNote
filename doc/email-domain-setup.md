# Email Domain Setup — `approachnote.com`

Setup date: 2026-05-04
Provider: **Google Workspace** (Business Starter plan)
Domain registrar / DNS host: **Squarespace**

This doc captures the complete email configuration for `approachnote.com` so it
can be reproduced, audited, or revisited if anything breaks.

---

## What we set up

- **Custom-domain email** at `approachnote.com` via Google Workspace.
- **Primary mailbox:** `dave@approachnote.com` (the actual user account).
- **Aliases on the same mailbox:** `support@`, `hello@`, `info@`, etc. — added
  as alternate addresses on the primary account in Google Admin → Users → your
  account → "Add alternate email." Aliases are free (Google allows ~30 per
  user) and route to the same inbox; outbound can be sent *as* any alias via
  Gmail → Settings → Accounts → "Send mail as."
- **Full deliverability stack:** SPF, DKIM, DMARC all configured.

## Architecture / who owns what

| Layer | Provider | What it does |
|---|---|---|
| Domain registration | Squarespace | Owns `approachnote.com` (registrar) |
| DNS hosting | Squarespace | Authoritative name servers; where all the records below live |
| Web hosting | Render | A records for `approachnote.com`, `www.`, `admin.`, `api.` |
| Email hosting | Google Workspace | MX-routed inbound mail; sends outbound from custom-domain mailboxes |
| Transactional email | SendGrid | Password-reset / system notifications (currently sends from a SendGrid-owned subdomain — see "Future" below) |

DNS for the website (A/CNAME records pointing the four hostnames at Render)
and DNS for email (MX + TXT records for SPF/DKIM/DMARC) live side-by-side at
Squarespace. They don't conflict — different record types.

## DNS records (canonical list)

All records below live at Squarespace under DNS settings for `approachnote.com`.

### MX records — inbound mail to Google Workspace

| Type | Name | Priority | Mail Server |
|---|---|---|---|
| MX | `@` | 1  | `ASPMX.L.GOOGLE.COM` |
| MX | `@` | 5  | `ALT1.ASPMX.L.GOOGLE.COM` |
| MX | `@` | 5  | `ALT2.ASPMX.L.GOOGLE.COM` |
| MX | `@` | 10 | `ALT3.ASPMX.L.GOOGLE.COM` |
| MX | `@` | 10 | `ALT4.ASPMX.L.GOOGLE.COM` |

**Why the 5-record legacy format and not the consolidated `smtp.google.com`?**
We initially used the newer `smtp.google.com` single-record MX (introduced
late 2023). It works for actual mail flow, but Google's own diagnostic
tools — including `toolbox.googleapps.com/apps/checkmx/` and, critically, the
DKIM verification flow in Google Admin — don't fully recognize it yet, which
manifested as "No Google mail exchangers found" warnings and prevented DKIM
activation. The legacy multi-MX format has been the canonical Google Workspace
setup for ~15 years and is universally recognized. **Stay on legacy** until
Google's tooling catches up.

### TXT records — deliverability

#### SPF (single TXT at root)
- **Name:** `@`
- **Value:** `v=spf1 include:_spf.google.com ~all`

`~all` is "softfail" — failing mail is marked suspicious but not rejected.
Right starter posture; tighten to `-all` (hard fail) once you're confident
nothing legitimate is missing from the includes.

**Important:** there can only ever be ONE SPF record per domain. When SendGrid
gets domain-authenticated against `approachnote.com` (see "Future"), update
this value rather than adding a second record:
```
v=spf1 include:_spf.google.com include:sendgrid.net ~all
```

#### DKIM
- **Name:** `google._domainkey`
- **Selector:** `google` (Google Workspace default)
- **Key length:** 2048-bit (chose stronger over the 1024-bit default)
- **Value:** the long `v=DKIM1; k=rsa; p=MIIB...IDAQAB` string generated in
  Google Admin → Apps → Google Workspace → Gmail → Authenticate email.

The DKIM record value is auto-generated by Google. If you ever need to
re-retrieve it, find it in Google Admin under the same path. The selector
stays `google` unless you regenerate (which would still keep `google` as the
selector but produce a new key).

#### DMARC
- **Name:** `_dmarc`
- **Value:** `v=DMARC1; p=none; rua=mailto:dave@approachnote.com; pct=100; adkim=s; aspf=s`

Field-by-field:
- `p=none` — monitor only, don't enforce. **Currently in this state.**
- `rua=mailto:dave@approachnote.com` — daily aggregate reports go here from
  every receiving provider (Google, Microsoft, etc.).
- `pct=100` — apply policy to 100% of mail (relevant once `p=` tightens).
- `adkim=s` / `aspf=s` — strict alignment (From-domain must exactly match
  DKIM/SPF domain, not a parent).

#### Google domain-verification (separate, kept for the record)
- **Name:** `@`
- **Value:** `google-site-verification=e23_ko8TNBVRA2-vWIwzb42zVH6_htWQtPj_Hmsw-uQ`

This proves domain ownership to Google and was added during the initial
Google Workspace signup. Do not delete it.

### A / CNAME records (web traffic, separate from email)

These are unrelated to email but listed for completeness so DNS edits don't
accidentally break the website. Maintained for the four web hostnames:
`approachnote.com`, `www.approachnote.com`, `admin.approachnote.com`,
`api.approachnote.com` — all pointing to the Render-hosted Flask service.
See `render.yaml` for the deployment config; nothing in this email setup
should ever touch those records.

## Verification commands

Quick sanity checks from the terminal:

```bash
# MX (should return all five legacy records)
dig MX approachnote.com +short

# SPF
dig TXT approachnote.com +short

# DKIM
dig TXT google._domainkey.approachnote.com +short

# DMARC
dig TXT _dmarc.approachnote.com +short
```

For deeper checks that bypass local-resolver caching, use Google's own dig
tool (it queries from Google's own infrastructure — closest to what their
internal verifiers see):
- DNS lookup: <https://toolbox.googleapps.com/apps/dig/>
- Mail-DNS lint: <https://toolbox.googleapps.com/apps/checkmx/>

For DKIM-specific parser validation: <https://easydmarc.com/tools/dkim-record-lookup>

## End-to-end smoke test

After all records are live, send a test email *from* a `@approachnote.com`
address *to* an external mailbox you control (a personal Gmail works). In the
received message, **Show original** (Gmail's three-dot menu). You should see:

```
SPF:    PASS  with IP <google IP>
DKIM:   'PASS' with domain approachnote.com
DMARC:  'PASS' with domain approachnote.com
```

If the DKIM line shows `gappssmtp.com` instead of `approachnote.com`, custom
DKIM is not yet activated in Google Admin — Google is signing with their
shared fallback key. Mail still authenticates and DMARC still passes (via SPF
alignment), but spam-filter trust is weaker until custom DKIM activates.

## Decisions worth remembering

1. **Path A over Path B.** Chose Google Workspace ($6–7/user/month) over a
   free aliasing service like ImprovMX. Worth it for a launching commercial
   product — proper sender reputation, real inbox-per-address (via aliases),
   Calendar/Drive on the domain, fewer "lands in spam" battles.

2. **Primary mailbox is `dave@`, not `support@`.** The primary username is
   the account's identity (login, file ownership, Calendar). Roles change,
   people don't — and `support@` can be promoted to a Google Group later
   without disrupting anything if/when the team grows. All role addresses
   (`support@`, `hello@`, `info@`) live as aliases on the same mailbox; Gmail's
   "Send mail as" picks the right From per conversation.

3. **Legacy multi-MX over consolidated `smtp.google.com`.** Even though
   Google publishes the consolidated record as their newer recommendation,
   their own internal tooling doesn't fully support it yet. Discovered this
   when DKIM activation refused to verify despite all DNS being correct —
   `checkmx` reported "No Google mail exchangers found" pointing at the
   single-record MX. Switching to the legacy 5-record set unblocked DKIM.

4. **DMARC starts at `p=none`.** Monitor mode for ~1 week of real sending
   to discover senders we forgot (forms, calendar invites, marketing tools)
   that need to be added to SPF first. Promotion path: `p=none` →
   `p=quarantine` (failing mail goes to spam) → `p=reject` (failing mail
   bounces). Don't rush — the whole point of `p=none` is to learn before
   enforcement breaks something legitimate.

## Troubleshooting log

### DKIM activation refused to verify (resolved 2026-05-04)

**Symptom:** Google Admin → Authenticate email → status stuck at *"Email
authentication was not verified. Please allow 48 hours for DNS to update..."*
even though `dig` from multiple resolvers (Google's `8.8.8.8`, Cloudflare's
`1.1.1.1`, Quad9's `9.9.9.9`) all returned the correct DKIM record. Waited
overnight, no change.

**Root cause:** The MX record was set to the new consolidated
`smtp.google.com` format, which Google's DKIM verifier doesn't (yet) treat
as a valid Google MX. `checkmx` confirmed this by reporting "Domain must
have at least one mail server" and "No Google mail exchangers found" —
despite `dig MX` returning the record cleanly. The DKIM verifier appears to
gate activation on whether it sees Google as the mail provider, and the new
format wasn't being recognized.

**Fix:** Replaced the single `1 smtp.google.com` MX record with the legacy
5-record set (`ASPMX.L.GOOGLE.COM` + 4 `ALT*.ASPMX.L.GOOGLE.COM`). Within
~10 min, `checkmx` flipped to green and DKIM verification went through on
the next click of START AUTHENTICATION.

**Lesson:** When Google's "newer" recommendation conflicts with their own
tooling, trust the tooling — pick the format that all their tools recognize.
Worth checking again in 12+ months whether the consolidated format has
become universally supported.

### DKIM activation refused to verify again (resolved 2026-05-05)

**Symptom:** Same "Email authentication was not verified" error, but this
time the legacy 5-record MX set was already in place and all DNS records
verified correctly from `8.8.8.8`. Multiple clicks of START AUTHENTICATION
failed over several hours. Google Admin Toolbox CheckMX timed out for the
domain (likely a transient tool issue unrelated to DNS).

**Root cause:** Unknown — DNS was correct. Google's internal DKIM verifier
appeared to have stale state for the existing key.

**Fix:** Clicked "Generate New Record" in Google Admin to produce a fresh
2048-bit key, replaced the `google._domainkey` TXT record in Squarespace
with the new value, confirmed propagation via `dig @8.8.8.8 TXT
google._domainkey.approachnote.com +short`, then clicked START
AUTHENTICATION. Verified immediately.

**Smoke test result:** Headers on a sent email confirmed:
`dkim=pass header.i=@approachnote.com header.s=google` — custom DKIM
active, no longer falling back to `gappssmtp.com`.

**Lesson:** If START AUTHENTICATION keeps failing despite correct DNS,
generate a new DKIM key rather than waiting. The regeneration appears to
force Google's verifier to re-check rather than rely on cached state.
